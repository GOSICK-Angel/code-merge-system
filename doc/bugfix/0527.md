系统层问题解决方案

  P0 — 正确性门禁（破损合并直接放行的根本缺口）

  方案 1：合并产物编译/类型/lint 终检门禁 ⭐最高优先

  根因：report_generation.py:_finalize_working_tree 直接落盘提交，全程无任何"产物能否构建"的检查。30 个 TS 错误一路走到 COMPLETED。

  方案——新增配置驱动的 verification 阶段（置于 judge_review 之后、report_generation 之前）：

  - 新增 config 块（MergeConfig）：
  verification:
    enabled: true
    commands:            # 留空则跳过；首跑由 setup 向导从 repo 自动探测
      - "tsc --noEmit -p packages/zod/tsconfig.json"   # 探测自 tsconfig
      - "pnpm -s lint"                                  # 探测自 package.json scripts
    timeout_seconds: 600
    on_failure: awaiting_human   # awaiting_human | fail | warn
  - 新增 src/core/phases/verification.py：在工作分支跑 commands，解析 stderr 的错误行，写入 state.verification_results。
  - 失败时进 AWAITING_HUMAN（复用现有人工闸口），把编译错误作为一个新的 DecisionPhase（build_failure）抛给用户：accept / abort / rerun（让 executor 带着错误信息重试）。
  - 目标仓库无关：commands 全部来自 config / 自动探测（package.json scripts.build|typecheck、tsconfig.json、pyproject 的 mypy/pytest），src/ 内无任何语言写死。

  代价：中（一个新 phase + config 字段 + setup 探测）。收益：一次性兜住重复声明、幻觉 API、跨文件类型不一致——本次全部 30+4 错误都会被它拦下。

  方案 2：重复符号确定性扫描（廉价、无需 LLM）

  根因：chunk_processor.merge_chunks 仅拼接分块结果、不做去重；相邻 chunk 的 LLM 合并各自"补全"了同一个顶层结构 → ZodNumberFormat/ZodIntersection/ZodSuccess/int/number/nonoptional 各重复一遍。

  方案——加入 judge 的确定性检查（与现有 O-J2/O-J3 同层）：
  - 新增 src/tools/duplicate_symbol_check.py：对合并后文件，按扩展名用正则提取顶层 export (const|function|interface|class) <name> / def <name> / class <name>，统计同名重复。
  - 复用 chunk_processor._boundary_pattern(ext) 的语言映射保持通用。
  - 任一重复 → 产出 JudgeIssue(severity=HIGH)，且不可被 meta-review 降级（见方案 5）。

  代价：低（纯静态、确定性、可单测）。这是分块合并 bug 的"安全网"，即使方案 3 没接也能兜住。

  P1 — Executor 合并质量

  方案 3：分块合并的接缝对账 + 幻觉符号护栏

  根因（已定位）：
  - _execute_chunked_semantic_merge 把每对 (curr_chunk, target_slice) 独立喂 LLM，结果直接 merge_chunks 拼接。LLM 不知道相邻 chunk 产出了什么 → 边界结构重复/撕裂。
  - executor 的 fidelity guard（docstring 自述）只抓"凭空造的字符"，不抓"缺内容"，也不抓"重复"；且无法验证新引入的跨模块符号（core._isoWeek）是否真实存在。

  方案：
  1. 接缝去重：merge_chunks 后增加一次顶层符号对账——若相邻 chunk 输出出现同名顶层声明，保留一个（优先含 upstream 改动的版本），其余删除。或更稳妥：分块合并完成后做一次整文件 reconcile pass（仅当文件 < 上下文上限时），把分块结果交给 LLM
  "去重 + 校对接缝"。
  2. 幻觉符号护栏（确定性）：合并产物中新出现的跨模块引用（如 core._isoWeek、core.$ZodISOWeek）必须能在仓库中解析到定义，否则升级人工并在提示里注明"analyst 已预警需新建 core._isoWeek"。这把 conflict_analyst 早就给出的预警变成 executor
  的硬约束。
  3. max_tokens 自适应：分块时按 chunk 预估输出 token 动态设上限（本次 4096 直接截断 json-schema-processors.ts），而非固定值。

  代价：中。优先做 (2) 幻觉护栏（确定性、收益最大），(1) reconcile 次之。

  方案 4：特性保留从"文件存在"升级到"符号存在"

  根因（已定位）：scar_list_builder 生成的校验是 CustomizationVerification(type="file_exists")——只查文件在不在，不查 fork 的具体新增是否还在。所以 regexes.ts 丢了 cidrv6Mapped 仍报 PASS（假 PASS）。

  方案：
  - 为 scar 自动生成 type="symbol_present" 的校验：从 base→fork diff 提取 fork 新增的顶层导出符号名（确定性 git+正则），合并后断言这些符号仍存在于对应文件。
  - 加一个总量指标 加性导出保留率 = 合并后存活的 fork-新增导出 / fork-新增导出总数，写入 merge_report 与 acceptance 门槛。
  - 纯新增符号丢失 → feature inventory 标 FAIL（而非 PASS），并阻断 COMPLETED 或转人工。

  代价：低-中。这是 fork 保护的核心，且确定性可测。

  P2 — Judge 与流程健壮性

  方案 5：Judge 不得浅检放行、不得自我否定可验证问题

  根因（已观察）：
  - "Judge skipped 1 high-confidence file (local syntax OK)" → 对 30 错误的 classic/schemas.ts 凭浅层语法检查跳过审查。
  - round 0 正确发现 json-schema 未整合 upstream，meta-review 却判 [downgraded: hallucinated evidence]（实际为真）。

  方案：
  1. "syntax OK" 的跳过条件升级：把"本地语法 OK"替换为方案 1/2 的真实编译/重复符号结果——确定性检查未过的文件禁止因 high-confidence 跳过。
  2. 可验证问题不可降级：凡 JudgeIssue 指向"文件 == 某一侧 / 符号缺失 / 重复声明"等确定性可核验的断言，meta-review 不能以"hallucinated"否决；先用确定性检查复核（working==fork? 符号在不在?），核实为真则保留。本次 json-schema 问题用 git diff
   一比即知 judge 是对的。
  3. 在 judge contract（src/agents/contracts/judge.yaml）里把这两类确定性证据作为 input，提示走 gate_registry。

  代价：中。与方案 1/2 共享确定性检查结果，边际成本低。

  方案 6：内部 escalate(0.00) 文件禁止绕过人工闸口/静默丢弃

  根因（已观察）：treeshake/*、no-circular-imports.test.ts 标 escalate_human 0.00 却没进我决策的 pending 列表，最终从树中丢失。human_review.py 收集 pending 的口径与 executor 内部 escalate 的口径不一致。

  方案：
  - human_review 收集 pending 时，纳入所有 confidence==0.0 / decision==escalate_human 的文件（含 commit-replay 阶段新增、不在初始 23 文件计划内的）。
  - finalize 前做一致性断言：任何 escalate/未决文件若未出现在人工闸口且未落盘，必须在 merge_report 显式列为"DROPPED (unresolved)"并阻断静默 COMPLETED。

  代价：低-中。主要是统一 pending 收集口径 + 一条 finalize 断言。

  方案 7：决策 YAML 畸形时必须报错（fail-loud）

  根因（已踩坑）：decisions_bundle.parse_bundle 仅当存在 version==2 才走 V2；带 rounds: 但漏写 version: 2 时静默当成空 V1 plan_review round，决策被丢弃且无任何报错（我首次 resume 即中招）。

  方案：parse_bundle 加显式校验——
  - 出现 rounds: 但缺 version: 2 → 要么自动识别为 V2，要么抛 ValueError（推荐：检测到 rounds 键即按 V2 解析，version 仅作可选标注）。
  - 解析后若该 round 匹配当前 phase 但 decisions/item_decisions 全空 → 打印显式 warning 而非静默。

  代价：极低（几行）。纯健壮性修复。
