## 3. 关键技术决策

### 决策 1：`diff.json` schema 设计

**结论**：在 `_schemas.py` 用 pydantic v2 定义 `DiffEntry` / `DiffReport`，与 procedure.md §3.2 字段一一对应，并显式扩展以下字段（仅追加，不修改文档示例字段）：

```python
class DiffEntry(BaseModel):
    sample_id: str
    category: Literal["A","B","C","D_MISSING","D_EXTRA","E"]
    loss_class: Optional[Literal["M1","M2","M3","M4","M5","M6"]] = None
    expected_human: bool
    system_decision: SystemDecision  # strategy/risk/human
    match: Literal["EXACT","SEMANTIC","MISMATCH"]
    label: Optional[Literal["MISS_UPSTREAM","MISS_FORK","WRONG_MERGE","EXTRA_NOISE"]] = None
    missed_lines: int
    extra_lines: int
    rationale_length: int            # 扩展：用于 RCR 指标
    discarded_content_present: bool  # 扩展：用于 DCRR 指标
    is_security_sensitive: bool      # 扩展：用于 SSER 指标
```

`DiffReport.meta` 含 `semantic_engine ∈ {"tree-sitter","fallback-bytes"}`、`generated_at`、`tier`、`run_set_sha`（runs 目录 sha256）。锚点：`procedure.md:104-125`。

### 决策 2：评估运行的隔离

**结论**：`run.py` 用以下三层隔离避免污染（procedure.md §6.1 / facts.md §3）：

1. **环境变量**：在子进程 env 中设 `HOME=<workdir>/home`、`MERGE_HOME=<workdir>/.merge`（如 `src/cli/main.py` 不识别此变量则 fallback 到 cwd 隔离），`ANTHROPIC_API_KEY=DUMMY-EVAL-KEY` 默认值（除非显式 `--use-real-keys`）。
2. **cwd 隔离**：每个 sample 的子进程 `cwd=<workdir>/runs/<sample_id>/working_tree`，子进程产生的 `.merge/` 全在该目录下。
3. **Memory wipe**：跑前删除 `<workdir>/home/.claude/projects/`（如存在）；跑后断言 `runs/<sample_id>/.merge/memory/` 不在 git 工作树外。
4. 若 `merge` CLI 不识别 `MERGE_HOME`：Phase 3 用 fallback——直接以 `cwd` + 临时 `HOME` 隔离即可，不为此修改 `src/`（facts.md §9 禁止）。

### 决策 3：DET / CPC 的 N 次跑：串行

**结论**：`consistency.py` 自身**不**触发 N 次跑，只读已有的多个 `runs/`。N 次跑由调用方 shell 循环（procedure.md §4 已示例）或 `run.py --seed N --workdir/run-N` 串行触发。

理由：

- 评估 `merge` 子进程内部已并发（每 sample 多 agent），外层再并发会导致 API rate-limit + cost 不可控；
- 串行可保证 `cost_usd / wall_time_seconds` 测量不受邻居跑影响（metrics.md §6.3 P50/P95 统计需要纯净计时）；
- 同 workdir 不同 `--seed` 不复用 cache（决策 4）。

`run.py --concurrency` 仅用于"同一次 run 内不同 sample 并发"，不用于 N 次重复 run。

### 决策 4：AST 等价比较，无新依赖

**结论**：`_ast_equiv.py` 实现优先级如下：

1. 尝试 `import tree_sitter` —— 成功则用 tree-sitter 解析、对比"去掉 comment / whitespace 节点"的 AST canonical 形式（仅在 optional `[ast]` extras 已装时可用）；
2. 失败则降级：先 normalize（去 BOM、统一换行 `\r\n→\n`、去尾随空白、去注释行——按文件后缀分派 `# / // / /* */ / <!-- -->`），再字节相等；
3. 非代码文件（`.json / .yaml / .toml`）走 normalize+sort（pydantic/json 标准库可读取）；
4. 二进制文件走严格字节相等（已在 RiskLevel.BINARY 中标记，原则上不进 F_eval）。

`DiffReport.meta.semantic_engine` 字段标注本次实际用了哪个引擎，避免把 fallback 结果误算成"AST 等价"。

锚点：`metrics.md §1.2`、`pyproject.toml:29-38` 确认 tree-sitter 是 optional。

### 决策 5：`lock.py` 用 sha256

**结论**：`sha256(b"\n".join(sorted_relpaths_then_file_sha256))`，**不**用 git blob hash。

理由：

- 评估样本可能含二进制 tar，git blob hash 需要 git 仓库上下文；sha256 自包含；
- 与 `acceptance.md §3` `tier1_lock_sha` 字段对齐 —— 该字段未规定算法，sha256 是工程默认；
- gitpython 仍只用于 Tier-2 replay（`_git_oracle.py`）。

`lock.json` schema：

```json
{
  "tier": 1,
  "version": "1",
  "samples": {
    "t1-0001": {
      "files": {"base.tar": "sha256:...", "meta.yaml": "sha256:..."},
      "sample_sha": "sha256:..."
    }
  },
  "lock_sha": "sha256:<concat of sorted sample_sha>"
}
```

### 决策 6：报告模板用 jinja2

**结论**：`eval_report.md.j2` 用 jinja2（已在 dependencies 中：`pyproject.toml:19`）。

理由：

- 字符串拼接对 procedure.md §3.1 六章节 + 失败案例 N 行表格不可维护；
- jinja2 已是生产依赖，不引入新依赖；
- 模板与代码分离，便于以后调整章节顺序而不改 Python 逻辑。

`gate.py` 输出的 `eval_acceptance_*.json` 是机器可读 schema，**不**用模板渲染（直接 `model_dump_json(indent=2)`）。

### 决策 7：`acceptance.md` 阈值的机器可读化

**结论**：在 `tests/eval/manifests/acceptance_thresholds.yaml` 维护一份与 `acceptance.md §1/§2` 同步的 yaml 阈值表，作为 `gate.py` 唯一阈值数据源。

理由：

- markdown 表格解析脆弱，任何排版变化会破坏；
- yaml 在 `lock.py` 中纳入校验：当 `acceptance.md` mtime 或内容 sha 变化但 `acceptance_thresholds.yaml` 未更新 → CI 报错（提示同步）；
- 不修改 `doc/evaluation/acceptance.md`（facts.md §9 禁止）。

实现细节：`lock.py --verify` 里加一个 `_assert_acceptance_sync()` helper，比对 acceptance.md 当前 sha 与 yaml 中记录的 `synced_with_sha` 字段。

### 决策 8：不引入新运行时依赖

确认所有第三方 import 来自 `pyproject.toml:9-23`：`pydantic / gitpython / unidiff / click / pyyaml / jinja2 / python-dotenv / tenacity`。`tree-sitter` 仅在 optional `[ast]` 中，且必须有 fallback。dev only：`pytest / pytest-asyncio / pytest-mock / pytest-cov / mypy / ruff / types-PyYAML`。

**禁止**新增任何 `requirements*.txt` 或 `pyproject.toml` 改动（`scripts/` 不打入 wheel，无需注册）。

## 5. 验收标准（与 procedure.md §2 命令清单逐项对应）

每条命令都需以"dummy data"跑通（即 Phase 1 创建的 1+1 个最小 sample），最终在 Phase 8 集成测试中作为单一 pytest 函数验证。

| 命令 | 验收点 | Phase |
|---|---|---|
| `python scripts/eval/lock.py --verify` | 退出码 0；篡改 sample 后非 0 | Phase 1 |
| `python scripts/eval/lock.py --update` | 写出 3 份 `*.lock.json`，含 `lock_sha` | Phase 1 |
| `python scripts/eval/prepare.py --tier 1 --out /tmp/eval-tier1` | workdir 内每 sample 有 `working_tree/` + `golden_tree/` + `meta.yaml` | Phase 2 |
| `python scripts/eval/run.py --tier 1 --workdir /tmp/eval-tier1 --concurrency 4 --merge-bin <fake> --merge-args "--no-web --ci"` | 每 sample 产 `runs/<id>/{run_meta.json, merge_state.json, plan_review.md, merge_report.md}` | Phase 3 |
| `python scripts/eval/diff_against_golden.py --runs ... --datasets ... --output /tmp/.../diff.json` | `diff.json` schema 符合 procedure.md §3.2 + 扩展字段 | Phase 4 |
| `python scripts/eval/summarize.py --diff ... --runs ... --output reports/eval_report_*.md` | report.md 含全部六章节 + 至少 9 个指标值 | Phase 5 |
| `python scripts/eval/gate.py --report ... --acceptance ... --baseline ... --output reports/eval_acceptance_*.json` | json 含 `gates/verdict`，退出码按决策 6 三态 | Phase 6 |
| `python scripts/eval/consistency.py --runs /tmp/eval-det --metric DET` | 输出 DET 数值 + 不一致样本表 | Phase 7 |
| `python scripts/eval/consistency.py --runs /tmp/eval-cpc --metric CPC` | 输出 CPC 数值 + 不一致样本表 | Phase 7 |

**全局验收**：

- `pytest tests/eval/ --cov=scripts/eval --cov-fail-under=80` 通过
- `mypy scripts tests/eval` 通过
- `ruff check scripts tests/eval` 通过
- `tests/eval/integration/test_e2e_tier1.py` 单测试 PASS（procedure.md §2 全链路）
- `grep -RE '(cvte|dify|insforge)' scripts/ tests/eval/datasets/ tests/eval/manifests/` 无匹配（facts.md §9 / CLAUDE.md "Project Generality"）
